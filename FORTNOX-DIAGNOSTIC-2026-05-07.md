# Fortnox Integration Diagnostic
> Generated by Claude Code on 2026-05-07
> Read-only investigation. No code changed.

## 1. Executive summary

The OAuth code path exists end-to-end (connect → callback → token exchange → encrypted storage → refresh), is exposed via a customer-visible "Connect" button on `/integrations`, and calls 2 Fortnox API endpoints (`/supplierinvoices`, `/invoices`) for monthly P&L sync. A separate, more sophisticated PDF-extraction pipeline (validators + AI auditor + line-item parser + apply chokepoint) is the primary data flow used in production for Vero and Rosali — the OAuth sync path coexists with it but is the simpler of the two and writes to `tracker_data` directly rather than through the canonical `projectRollup` writer. The `FORTNOX_CLIENT_ID` / `FORTNOX_CLIENT_SECRET` env vars are referenced but the diagnostic cannot tell whether real values are populated in production or whether the developer-account application has been approved by Fortnox. Of the ~233 endpoints in the v3 OpenAPI spec, the integration code calls 2 in the sync path, 8 in a separate discovery/analysis tool that does not feed customer data flows, and zero of the financial-year, voucher, account, or salary endpoints despite the `salary` and `bookkeeping` scopes being requested. A Fortnox-API-first onboarding for a new customer is structurally possible today (OAuth path is wired) but the data depth captured by sync is materially shallower than the PDF path, and the disconnect flow does not delete Fortnox-derived data.

## 4. OAuth and credentials

- Client credentials: `process.env.FORTNOX_CLIENT_ID`, `process.env.FORTNOX_CLIENT_SECRET`. `.env.example` carries placeholders only. Whether real values are set in production is not derivable from code.
- Scopes requested at connect: `bookkeeping invoice supplierinvoice salary companyinformation` (single space-joined string, line 55-61 of `app/api/integrations/fortnox/route.ts`).
- Tokens stored in `integrations` table: `credentials_enc` (encrypted JSON of `{access_token, refresh_token, expires_at}`), `token_expires_at` (timestamptz), keyed by `(business_id, provider)` with `org_id` carried alongside.
- Encryption: `lib/integrations/encryption.ts` (read by import; not opened in this diagnostic).
- Refresh flow: triggered lazily inside `syncFortnox` when `expires_at - Date.now() < 60 minutes`. Also exposed as `POST { action: 'refresh' }` for explicit calls. Both call `POST /oauth-v1/token` with `grant_type=refresh_token` and HTTP Basic Auth.
- Rolling refresh tokens: handled — `refresh_token: new_refresh ?? refresh_token` (line 533) preserves the old token if Fortnox does not issue a new one.
- On refresh failure: sets `integrations.status='error'` with `last_error='Token refresh failed — please reconnect Fortnox'`. No customer notification fires (no email, no Sentry custom event identifiable from this file).
- No scheduled refresh cron. Refresh only happens on user-triggered sync. If a customer doesn't sync for an extended period, the access token expires silently; the refresh attempt only runs at the next sync trigger.
- Health-check cron (`app/api/cron/health-check/route.ts:178`) issues an unauthenticated `fetch('https://api.fortnox.se/3/')` for `case 'fortnox':`. The "real" per-customer call is in a comment immediately above. The result of this base-URL ping informs `health` status.
- CSRF protection: `state` parameter is HMAC-SHA256-signed with `ADMIN_SECRET` over `{orgId, businessId, nonce}`. Verified in `verifyState` with timing-safe equality. Rejected callbacks redirect to `/integrations?error=fortnox_invalid_state`.
- Admin audit: `recordAdminAction(ADMIN_ACTIONS.INTEGRATION_ADD)` is called on successful callback with `actor='oauth_callback'`.

## 6. Production readiness

Stated as factual code-level findings.

- `// @ts-nocheck` is present on `app/api/integrations/fortnox/route.ts` (line 1) — the entire OAuth + sync file is excluded from TypeScript checking.
- `// @ts-nocheck` is present on `lib/fortnox/resultatrapport-parser.ts` (line 1).
- `console.error` calls in OAuth callback paths (lines 141, 153, 181, 187, 220) emit to Vercel logs only; no structured `log.error` and no Sentry capture observed in this file.
- No hardcoded organisation IDs in the Fortnox-specific code paths.
- No sandbox vs production URL toggle. `FORTNOX_API`, `FORTNOX_AUTH`, `FORTNOX_TOKEN` are production URLs hardcoded as module constants.
- Pagination handling in the sync function: not implemented. `GET /supplierinvoices?fromdate=&todate=` and `GET /invoices?fromdate=&todate=` parse `SupplierInvoices` / `Invoices` arrays from a single response. If the response is paginated by Fortnox (default page size 500 in v3) and a busy business exceeds it for a single month, additional records are silently missed.
- Idempotency on writes: `tracker_data` upsert uses `onConflict: 'business_id,period_year,period_month'`. Repeated sync calls for the same period overwrite; no version stamping.
- Rate-limiting awareness: per-user `rateLimit(auth.userId, { windowMs: 60 * 60_000, max: 20 })` on the sync endpoint. Fortnox's 25 req / 5 sec server-side limit is not explicitly respected; the sync makes 2 parallel API calls per invocation so the headroom is intact for typical usage.
- Error-swallowing patterns: the OAuth route catches token-exchange failures (lines 178-189) and redirects with an `error=fortnox_*` query param; the customer sees a query-string error code without a user-readable message. The sync function sets `last_error` but does not page anyone.
- Timeout on health-check: 8 s `AbortController` (line 171). No timeout on the OAuth sync function's two parallel API calls — the request will hang up to the function's `maxDuration` (not set on this route, defaulting to platform timeout).
- Background sync after callback: `syncFortnoxInBackground(orgId, access_token).catch(console.error)` is fire-and-forget (line 234). Not wrapped in `waitUntil` so on Vercel the function may terminate before the sync completes if the lambda is recycled.
- Two writers to `tracker_data` exist: `app/api/integrations/fortnox/route.ts::syncFortnox` (lines 442-457, writes with `source='fortnox'`) and `app/api/fortnox/apply/route.ts` via `lib/finance/projectRollup.ts`. CLAUDE.md states `projectRollup` is the only function that turns Fortnox extractions into `tracker_data` rows; the OAuth sync writer predates that invariant and is a second writer.
- Classification drift: BAS account → category mapping appears in two places — a 30-entry hardcoded map at `app/api/integrations/fortnox/route.ts:365-376` (used by OAuth sync) and the broader range-based map in `lib/fortnox/classify.ts::classifyByAccount` (used by PDF parser + AI extractor). These can drift.
- Line 413: `if (category === 'alcohol') food_cost += amount` — the OAuth sync rolls alcohol into food_cost. The PDF path keeps `alcohol_revenue` and `alcohol_cost` separate per VAT classification (`lib/fortnox/classify.ts::classifyByVat`).

## 7. Legal / contractual signals

These are code-level observations, not legal conclusions.

- `app/privacy/page.tsx`, `app/terms/page.tsx`, `app/security/page.tsx` exist as rendered routes. Whether their content addresses Fortnox specifically as a sub-processor is not inspected here.
- OAuth flow shows Fortnox's own consent screen (their hosted authorisation page). The redirect target is `https://apps.fortnox.se/oauth-v1/auth` with `response_type=code` and `access_type=offline`. CommandCenter does not display its own consent screen before redirecting.
- Disconnect flow (`app/integrations/page.tsx:281-290`):
  - Updates `integrations` row to `status='disconnected'` and nullifies `credentials_enc`.
  - Does NOT delete `fortnox_uploads`, `tracker_data` rows derived from Fortnox, `tracker_line_items`, or `extraction_jobs`. These remain in the database after disconnect.
  - Uses the user's session client (`createClient()`) and relies on RLS for org-scoping. The `.update()` call filters only by `provider='fortnox'`, not by `org_id`.
- No "data export" or "right to be forgotten" flow specific to Fortnox-derived data observed in `app/api/fortnox/*` or `lib/fortnox/*`. The hard-delete tenant-table list referenced in CLAUDE.md cascades into `ai_*` tables; whether `fortnox_uploads` and `tracker_*` are included is not visible from the Fortnox-specific code reviewed here.
- Token storage: encrypted at rest via `lib/integrations/encryption.ts` (read by import; not opened in this diagnostic). HMAC state signing prevents an attacker from binding their Fortnox account to a victim's CommandCenter org via the callback.
- `LEGAL-OBLIGATIONS.md` is the explicit legal source-of-truth file referenced in user memory; not read by this diagnostic.
