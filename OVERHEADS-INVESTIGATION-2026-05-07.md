## 1. Flag types

The schema-level enum is in `archive/migrations/M039-OVERHEAD-REVIEW.sql:80-86`:

```sql
flag_type CHECK (flag_type IN (
  'new_supplier',
  'price_spike',
  'dismissed_reappeared',
  'one_off_high',
  'duplicate_supplier'
))
```

Five values are valid in storage. Only **four** are written by the rule engine. The fifth (`duplicate_supplier`) is reserved in the schema and the UI's `FLAG_TONE` map (`app/overheads/review/page.tsx:51-57`) but no code path produces it — orphaned from a planned-but-not-shipped detection rule.

All classification happens in `lib/overheads/review-worker.ts` (`detectForCategory`, lines 97-265). One pass per `(business_id, period_year, period_month, category)` tuple, fully deterministic, no AI.

| Name | Storage | Definition | Code |
|---|---|---|---|
| `new_supplier` | `overhead_flags.flag_type='new_supplier'` | First time this normalised label appears. Two sub-cases: (a) zero history AND amount < 5% of monthly category total → `'First time this line has appeared.'`; (b) recurring spend with history but no classification yet → `'Recurring spend not yet reviewed.'` | `review-worker.ts:204-221` |
| `price_spike` | `overhead_flags.flag_type='price_spike'` | Two paths: (a) UNCLASSIFIED supplier with history where `|amount - prior_avg| / prior_avg > 30%` AND `|delta| ≥ 1500 kr` → "Volatile cost: 12-mo avg X vs Y now"; (b) CLASSIFIED-essential supplier where `amount > baseline × 1.15` AND `delta ≥ 1500 kr` → "Up N% vs baseline (X kr)". | `review-worker.ts:212-229` |
| `dismissed_reappeared` | `overhead_flags.flag_type='dismissed_reappeared'` | Owner previously dismissed (`overhead_classifications.status='dismissed'`); supplier still appears in this period's books at any non-zero amount above the 500 kr noise floor. → "Was marked plan-to-cancel; still in books at X kr/mo." | `review-worker.ts:233-235` |
| `one_off_high` | `overhead_flags.flag_type='one_off_high'` | UNCLASSIFIED, NO history at all, AND amount ≥ 5% of monthly category total. → "One-off charge ≥5% of overheads, no prior history." | `review-worker.ts:205-207` |
| `duplicate_supplier` | (never written) | UI tone definition only at `app/overheads/review/page.tsx:56`. No classifier path produces it. Reserved in schema CHECK constraint. | — |

**Floor thresholds applied across all rules:** `MIN_FLAG_AMOUNT_SEK = 500` (no noise on tiny lines), `MIN_VOLATILE_DIFF_SEK = 1500` (volatility flags require absolute movement matters too).

**Storage shape per flag** (`overhead_flags`, full column list per M039 schema): `id`, `org_id`, `business_id`, `source_upload_id` (FK to fortnox_uploads, ON DELETE CASCADE), `line_item_id` (FK to tracker_line_items, ON DELETE CASCADE), `supplier_name`, `supplier_name_normalised`, `category` ('other_cost' | 'food_cost', M041), `flag_type`, `reason`, `amount_sek`, `prior_avg_sek` (null for new_supplier / one_off_high), `period_year`, `period_month`, `surfaced_at`, `resolution_status` ('pending' | 'accepted' | 'dismissed' | 'deferred'), `resolved_at`, `resolved_by`, `defer_until`, `ai_explanation`, `ai_confidence`. Idempotency key: `UNIQUE (business_id, source_upload_id, supplier_name_normalised, flag_type)`.

**Source data:** `tracker_line_items` (this period's lines + 12-month history in the same category), `overhead_classifications` (existing per-supplier decisions), `fortnox_uploads` (only as FK target; ON DELETE CASCADE handles supersede cleanup automatically).

**For Vero specifically:** I cannot enumerate the actual flag-type counts without DB access. The 27-flag count visible in the UI screenshot is the sum of `resolution_status='pending'` rows for Vero's `business_id`. A `SELECT flag_type, COUNT(*) FROM overhead_flags WHERE business_id='0f948ac3-...' AND resolution_status='pending' GROUP BY 1` query would expose the per-type breakdown.

## 2. AI explanations

**Trigger paths:**
1. **Eager pass after detection** — `lib/overheads/review-worker.ts::runExplanationPass` (line 268) runs synchronously at the end of every `runOverheadReview()`. Pulls all `resolution_status='pending'` flags for that period with `ai_explanation IS NULL` (max 50 in one call) and writes the result back via UPDATE. Best-effort: if the AI call fails or times out, the flags stay explanation-less and the rule reason remains the only context. Re-running the worker on the same period only re-fills NULLs (won't overwrite existing explanations).
2. **On-demand re-explain** — `POST /api/overheads/explain/[flagId]` (`app/api/overheads/explain/[flagId]/route.ts`, 165 lines after my Phase 3 edits yesterday). Triggered by the "re-explain" link inside the flag card's body. Single-flag pass with up to 18 months of supplier history + benchmarks + warm-cache invoice context (the cache enrichment landed yesterday in commit `e789569`). Overwrites the existing explanation.

**Endpoint or function** producing the explanation: `lib/overheads/ai-explanation.ts::explainOverheadFlags` (204 lines). Single Sonnet 4.6 call per batch.

**Model:** `AI_MODELS.ANALYSIS` = Sonnet 4.6. `max_tokens: 1500`. Forced `tool_choice: { type: 'tool', name: 'submit_flag_explanations' }` so output is always structured. 45 s `AbortController` timeout.

**Cache:** persisted on the row itself.

```sql
overhead_flags.ai_explanation TEXT
overhead_flags.ai_confidence  NUMERIC(3,2) CHECK 0..1
```

No separate cache table, no TTL — the explanation is treated as a piece of the flag's permanent state until the owner re-explains. The `runExplanationPass` worker uses `.is('ai_explanation', null)` so it won't overwrite. The re-explain endpoint unconditionally overwrites.

**Cost per call:** logged via `logAiRequest()` in `ai-explanation.ts:174-182` — input/output tokens recorded in `ai_request_log`. The header comment estimates **~$0.05 per business per period** (single Sonnet call with up to 50 flags batched). The on-demand re-explain is a single-flag call so cheaper — probably ~$0.01-0.02 per click.

**Latency:** observable from `ai_request_log.duration_ms`. The 45 s timeout caps the worst case. Typical Sonnet 4.6 with 1500 max_tokens + tool_use is 8-15 s; faster on small batches. I cannot quote a Vero-specific recent example without DB access.

### Direction 2 detail-pane question

**If the new design renders an AI explanation alongside every flag in a detail pane, would that require eager generation, a different caching strategy, or just an automatic fire-on-row-select?**

Short answer: **the existing model already covers it — no new infrastructure needed.**

- Eager generation already happens at detection time (`runExplanationPass` after `runOverheadReview`). Almost every flag in the database will already have an `ai_explanation` populated by the time the user sees it.
- The explanation is on the row, so the read path is free — the existing `/api/overheads/flags` endpoint already returns `ai_explanation` and `ai_confidence` in the row payload. Zero extra calls to render in a detail pane.
- The on-demand re-explain is the existing manual-trigger path for "I want a fresh take" and continues to work.

Edge case: a flag whose detection happened before PR 4 (2026-04-something) shipped, OR a flag where the AI call failed at detection time. These have `ai_explanation IS NULL`. The page already handles this — UI shows a "Generate AI explanation" button which fires the re-explain endpoint. Zero change required.

So: detail pane reads `ai_explanation` from the row payload directly; no fetch-on-select needed; the existing on-demand button covers the gaps. **Ship the UI redesign without touching the AI side.**

## 3. Decision state

**Endpoint:** `POST /api/overheads/flags/[id]/decide` (`app/api/overheads/flags/[id]/decide/route.ts`, 145 lines).

**Body:** `{ decision: 'essential' | 'dismissed' | 'deferred', reason?: string (max 1000 chars) }`.

| Decision | Classification table write | Flag row write | Scope |
|---|---|---|---|
| `essential` | UPSERT into `overhead_classifications` with `status='essential'`, `baseline_avg_sek = current period's amount`, `decided_by`, `decided_at`, `reason`. Conflict key: `(business_id, supplier_name_normalised, category)`. | UPDATE every pending row matching `(business_id, supplier_name_normalised, category)` to `resolution_status='accepted'`, `resolved_at`, `resolved_by`. | **Supplier-wide × category-wide × all-periods.** Single click resolves every pending row for that supplier in that category, in this business. Future periods stop flagging unless price > baseline × 1.15. |
| `dismissed` | Same as essential but `status='dismissed'`. Drives the "savings projection" (the dismissed amount counts as "I plan to cancel"). | Same bulk update pattern, `resolution_status='dismissed'`. | Same supplier-wide × category-wide × all-periods scope. Future re-appearance produces a `dismissed_reappeared` flag. |
| `deferred` | NO classification write. | UPDATE only the specific flag with `resolution_status='deferred'`, `defer_until = NOW() + 30 days`. | **Single-flag, per-instance.** The same flag will surface again in 30 days via `lib/overheads/expire-deferred.ts`'s sweep, which `/api/overheads/flags` calls before every read. |

**Reversibility:** the endpoint is **idempotent** — re-deciding the same flag overwrites the classification (decided_by/at update; reason replaced) and the resolution_status changes. Owner changing their mind from "dismissed" to "essential" works cleanly. **There is no audit trail beyond `decided_by`/`decided_at` on the classification row** — the previous decision is overwritten in place. If the owner changes from essential to dismissed back to essential, only the latest write is preserved.

**Effect on display:** `/api/overheads/flags` defaults to `resolution_status='pending'` only. Resolved flags are hidden by default; visible via `?include_resolved=1`. The UI doesn't currently surface the include_resolved flag — there's no "show decided" button on `/overheads/review`. Once a decision is made, the row vanishes from the queue.

**Audit-log feed:** none of the decisions write to `admin_audit_log` (that's reserved for admin-side actions). Customer decisions live only on the `overhead_classifications` and `overhead_flags` rows themselves.

## 4. Cross-period grouping

**Where computed:** `app/overheads/review/page.tsx:222-248` (`grouped` `useMemo`). **Client-side only**. The API returns flat flag rows; the page groups them in the browser.

**Grouping key:**

```ts
const key = `${f.supplier_name_normalised}::${cat}`
```

— normalised supplier name combined with category. M041 (added 2026-04-something) made decisions category-scoped, so a supplier appearing in BOTH `food_cost` and `other_cost` surfaces as **two separate cards**, with two independent decisions. Edge case in §74 of the prompt is correctly answered: kept separate, by design.

**"Latest" selection:** the `(year × 100 + month)` key picks the most recent period as the displayed flag; the rest go into `others[]` — that's what produces the "Also flagged in Feb 2026, Jan 2026, Mar 2025" footer.

### "Applies to all"

**The classification write is supplier-wide.** Decision endpoint (`flags/[id]/decide/route.ts:79-95`) writes a single `overhead_classifications` row keyed on `(business_id, supplier_name_normalised, category)` — no per-period rows.

**The flag-side is bulk UPDATE.** The same endpoint (lines 116-138) does:

```ts
.from('overhead_flags')
.update(update)
.eq('org_id', auth.orgId)
.eq('business_id', flag.business_id)
.eq('resolution_status', 'pending')
.eq('supplier_name_normalised', flag.supplier_name_normalised)
.eq('category', flag.category ?? 'other_cost')
```

— resolves EVERY pending row for that supplier-in-that-category, across all periods. One transaction. The endpoint returns `flags_resolved: N` so the UI can show "resolved 4 pending periods" feedback.

**Edge case (same supplier in two categories):** kept separate. Two independent classification rows, two independent bulk resolutions. Owner could mark "Konsultarvoden" essential as `other_cost` while flagging the same Swedish label as `food_cost` (rare but possible if accountant books a consultancy invoice partially against food).

## 5. Invoice drill-through

This was substantially shipped yesterday (2026-05-07, commits `9c2b7e4` + `e789569`). Current state:

**Component:** `app/overheads/review/page.tsx::InvoiceDrilldown` (lines 585-720). Inline expansion under each flag card — clicking "Show invoices ▾" expands a section in-place. Not a separate page, not a modal.

**Data source:** lazy fetch on first expand. POST to `/api/integrations/fortnox/drilldown` with `{ business_id, year, month, category }`. The endpoint:
- Cache check on `overhead_drilldown_cache` (M051) keyed by `(business_id, year, month, category)`, 5-min TTL
- Cache miss: parallel-fetches Fortnox vouchers (with rows) + supplier invoices for the period; joins by `VoucherSeries+VoucherNumber`; filters voucher rows by BAS account range matching the requested category; groups by supplier
- Returns `{ flagged_total, suppliers: [...], manual_journals: [...], fetched_at }`

**Per-invoice fields available** (from the drilldown response shape, see `app/api/integrations/fortnox/drilldown/route.ts:73-91`):

| Field | Source | Use |
|---|---|---|
| `source_type` | `'supplier_invoice'` or `'manual_journal'` | UI distinction |
| `source_id` | Fortnox `GivenNumber` (supplier invoice) or `series-number` (voucher) | Display + dedup |
| `fortnox_url` | `https://apps.fortnox.se/supplierinvoice/{GivenNumber}` or `/voucher/{series}/{number}` | "In Fortnox ↗" link |
| `file_id` | Fortnox `SupplierInvoiceFileConnections[0].FileId` | Drives the "View PDF" button → streamed via `/api/integrations/fortnox/file` proxy |
| `date` | ISO `YYYY-MM-DD` | Chronological sort |
| `invoice_number` | Display number (`InvoiceNumber` or `GivenNumber`) | Row label |
| `supplier_name` | From the matched Supplier Invoice row | Always same as the flag's supplier (filtered) |
| `amount` | Sum of voucher rows in the requested category × this invoice (kr) | Per-invoice contribution |
| `full_total` | Full Fortnox `Total` on the invoice (could be partial vs `amount`) | Informative — owner sees the full invoice value even when only part falls in this category |
| `account` | BAS account number (e.g. 5410) | Helps owner identify what was bookkept |
| `account_description` | Free text from Fortnox | "Förbrukningsinventarier" etc. |
| `description` | `Comments` from invoice | Optional context |
| `voucher_series`, `voucher_number` | The bookkeeping voucher reference | Used for non-invoice manual journals |

**Quantity:** per the drilldown's filter (rows in this supplier × in the matched account range × in this period), typically **3-15 invoices per flag** for an active vendor like Menigo or Lokalhyra. A flag for "IT-tjänster" with 4 flagged periods would NOT show 4× as many invoices on a single click — the drilldown is **scoped to the flag's period**, not the supplier's whole history. Each period's drilldown is a separate fetch.

### Direction 2 inline-detail-pane question

The current data shape **directly supports** rendering the supplier's invoices for the *currently-selected* period inline. The drilldown endpoint already returns the structured per-invoice list; the React state (`drilldown`, `matchedGroup`) already holds the expansion data per flag.

What would need to change for "always-visible in detail pane":
- **Auto-fire on selection** — currently the user has to click "Show invoices ▾". A detail-pane redesign would fire the fetch on flag-card-select instead. Same endpoint, same data, just a different trigger.
- **Multi-period spanning** — if the detail pane wants to show "all invoices across all periods this supplier was flagged" (not just the current period), that's a NEW shape. The drilldown endpoint is single-period; spanning would need either multiple fetches in parallel (one per period in `latestKey + others.map(o => o.periodKey)`) or a new endpoint that takes a date range.

The single-period default matches the current grouping in the UI (the latest flag is shown, others are listed by date). For Direction 2, "all periods at once" might be what you want; flagging this as a small build-or-not decision.

### Cron behaviour

**No cron writes to `overhead_flags`.** Detection runs only when triggered:

- `runOverheadReview()` is called from **two places only** — both inside `app/api/fortnox/apply/route.ts` (lines 250 and 378), wrapped in `waitUntil()` so the apply response returns immediately. Multi-period applies fire it sequentially per period; single-period applies fire it once unless the upload is `pnl_annual` (annual uploads write line items but not `tracker_data`, so the worker is skipped).
- `expireDeferredFlags()` runs at the top of every `/api/overheads/flags` GET (read-time sweep), reviving flags whose 30-day defer has expired.
- The AI explanation pass runs synchronously inside `runOverheadReview` after detection completes.

**Implication for Saturday's onboarding:** Vero's tracker_data populates from BOTH the existing PDF-applied months AND the new 12-month API backfill. **The API backfill writes `tracker_data` directly via `projectRollup`, bypassing `/api/fortnox/apply` and therefore NOT triggering `runOverheadReview`.** Result: Vero's `/overheads/review` page after Saturday's connect will show flags only for periods that had a PDF applied historically — not for the freshly-backfilled API months. This is consistent with the M050 worker's design (skipping months where PDF apply already populated, per the agreed PDF-priority invariant from Session 17), but it's worth knowing that API-derived months won't generate flags until either a PDF apply lands for that month OR the worker is wired into the API path. Surface but not within this investigation's scope to fix.

### Mobile responsiveness

The `/overheads/review` page is desktop-first. Specific concerns at a 380 px viewport:

- The flag-card grid (Title row: supplier + tone badge + category badge + period label) wraps via `flexWrap: 'wrap' as const` on the badge container — that part survives narrow widths.
- The right-aligned amount + period (page.tsx:509-514) collapses cleanly because the parent flexbox has `gap: 12` and `alignItems: 'flex-start'`.
- The action button row (`Defer 30d` / `Plan to cancel` / `Essential`, line 517) `justifyContent: 'flex-end'` — at 380 px, three buttons fit but barely.
- The drilldown invoice rows (lines 730+) use a hardcoded `width: 60` and `width: 140` for date/invoice-number columns — these will overflow / horizontal-scroll on mobile rather than wrap.
- The headline + filter buttons render fine.

Nothing is broken; nothing is delightful on mobile. The drill-down's per-row layout is the most likely candidate for mobile-specific tightening.

## Summary — gap from current state to Direction 2 design

**Pure UI redesign with two small data-side additions.** Specifically:

1. The list-of-cards → list+detail pane refactor is **purely visual** — every field the detail pane needs (`ai_explanation`, `reason`, `amount_sek`, `prior_avg_sek`, `period_year/month`, `supplier_name`, `category`, `flag_type`, `dept_breakdown` via the existing drilldown) is already in the data the page fetches today.
2. **One small new endpoint** — `/api/overheads/supplier-history` — for the 12-month per-supplier price-history chart. ~80 lines, single query against `tracker_line_items` already grouped by normalised supplier. Reuses the existing classifier helpers in `lib/overheads/normalise.ts`.
3. **One small client-side aggregation** — count of `flag_type='price_spike'` and `flag_type='dismissed_reappeared'` from the existing flags response for the headline strip; plus a new `?stats=1` parameter on `/api/overheads/flags` (or a new endpoint) that returns "decided in last 90 days" as a count.

What is **NOT** required: no schema migration, no AI architecture change, no new cache, no agent-level work, no changes to the classifier, no changes to the decision flow, no changes to invoice drill-through. Existing behaviour is sufficient.

The riskiest unknown is **mobile layout** — the current page is desktop-first and the detail-pane redesign at narrow widths needs a coherent collapse story (likely "list view" stays full-width and clicking a flag pushes a full-screen detail view, mobile-only). That's a UX decision, not a tech blocker.

## Recommended implementation-prompt shape

A **single visual-update prompt**, with two small data-side appendices that can be built first or alongside without splitting into two PRs. Both appendices are self-contained and ~half a day of work each: (a) `GET /api/overheads/supplier-history` for the chart, and (b) a `decided_last_90d` count on the flags endpoint or a small dedicated `/api/overheads/stats` route. The visual redesign itself is the biggest piece by far and shouldn't be split — the components are interlinked enough that piecing them across two PRs would create awkward intermediate states.
