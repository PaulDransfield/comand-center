**⚠ Flag — query parameter in redirect URI.** Fortnox's documentation says: *"A redirect URI must not contain parameters such as user authentication or hashtag (#)."* It is unclear from that wording whether routing parameters like `?action=callback` are also rejected, or whether the rule only forbids user-supplied auth state. Note: the OAuth flow uses an HMAC-signed `state` parameter for CSRF protection (built in `signState`, lib lines 33-37), which is the standard mechanism for what the documentation language is presumably forbidding; the `action=callback` parameter is a routing artefact, not auth state.

## 3. What the code does NOT yet do

Plain observations on the rest of the OAuth flow's state.

- **The OAuth flow has never been run end-to-end against a real Fortnox account.** Confirmed via DB check on 2026-05-07: zero rows in `integrations` have `provider='fortnox'` across the entire production database. See `FORTNOX-VERIFICATION-REPORT-2026-05-07.md` for the SQL and the broader implications.

- **No customer currently has Fortnox OAuth tokens stored.** Same finding — the table column the encrypted token would land in (`integrations.credentials_enc`) has no Fortnox-provider rows at all.

- **`/integrations` page — built.** `app/integrations/page.tsx` exists and is fully wired. It renders Fortnox as one of 14 provider cards (lines 19-95 are the static manifest), and the Connect button at line 456 calls `connectFortnox(businessId)` which redirects the browser to `/api/integrations/fortnox?action=connect&business_id={id}`. The page is rendered behind the standard `AppShell` auth gate.

- **Callback handler — implemented.** `app/api/integrations/fortnox/route.ts:122-238` (`handleCallback`) is the complete authorisation-code-grant exchange:
  1. Verifies the HMAC-signed `state` parameter (CSRF gate, line 151).
  2. Posts the code to `https://apps.fortnox.se/oauth-v1/token` with HTTP Basic auth using `FORTNOX_CLIENT_ID:FORTNOX_CLIENT_SECRET` (line 165).
  3. Encrypts the token JSON via `lib/integrations/encryption.ts` (AES-256-GCM with `CREDENTIAL_ENCRYPTION_KEY`).
  4. Upserts into `integrations` with `(business_id, provider='fortnox')` as the conflict key, storing `credentials_enc`, `token_expires_at`, `status='connected'` (line 203).
  5. Records an admin audit row via `recordAdminAction(ADMIN_ACTIONS.INTEGRATION_ADD)` (line 224).
  6. Fires a background sync via `syncFortnoxInBackground(orgId, access_token).catch(console.error)` (line 234).
  7. Redirects the customer back to `/integrations?connected=fortnox`.
  
  Token storage table is `integrations` (the existing multi-provider integration row), not a dedicated `fortnox_tokens` table. Nothing reads `fortnox_tokens` in the codebase; the prompt's mention of that table appears to assume a different schema than what's actually in production.

- **Token refresh flow — implemented twice.** Two implementations of effectively the same logic:
  - `app/api/integrations/fortnox/route.ts:484-547` (`refreshToken`) — exposed as `POST /api/integrations/fortnox` with `{action:'refresh'}` and called lazily inside `syncFortnox` when the token has < 60 min remaining (line 289).
  - `lib/sync/engine.ts:465-512` (`ensureFreshFortnoxToken`) — called inside the cron-triggered `syncFortnox` (lib/sync/engine.ts:515) when the token has < 5 min remaining.
  
  Both POST `https://apps.fortnox.se/oauth-v1/token` with `grant_type=refresh_token` and HTTP Basic auth. Both persist the new token via `lib/integrations/encryption.ts::encrypt`. Both honour Fortnox's rolling-refresh-token behaviour (use the new refresh token if returned, else keep the old one). Neither runs on a schedule — refresh only happens at sync time. If a customer hasn't synced for an extended period and their refresh token expires due to inactivity, the recovery path is reconnect (no proactive notification).
