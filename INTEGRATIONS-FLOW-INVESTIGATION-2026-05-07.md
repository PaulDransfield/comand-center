## 1. Page structure

`/integrations` is a **per-provider grid, not per-business**. The page renders one card per integration (14 cards), iterated from the static `PROVIDERS` array at `app/integrations/page.tsx:19-95`. Each card renders inside a single `.map()` loop at line 351 (`{PROVIDERS.map(provider => { â€¦ })}`).

The page reads ONE business at a time â€” whichever the user has selected in the sidebar:

- `selectedBiz` is initialised from `localStorage.getItem('cc_selected_biz')` at lines 109-111.
- A `useEffect` at lines 116-136 listens to `storage` events (line 102), so when the sidebar business switcher changes the active business, `/integrations` re-renders against the new selection.

Per-card connection state is computed inline (lines 353-360):

```tsx
const allIntegs = (integrations['__all__'] ?? []).filter(i => i.provider === provider.key)
const bizInteg  = selectedBiz
  ? allIntegs.find(i => i.business_id === selectedBiz)
  : allIntegs.find(i => !i.business_id) ?? allIntegs[0]
const integ  = bizInteg ?? null
const status = integ?.status ?? 'not_connected'
```

So a card's "Connected / Not connected" pill depends on `selectedBiz` â€” but the connect *action* on the same card has to follow through and write a row tied to that same `selectedBiz`. PK does. Fortnox doesn't.

There are also two buttons-per-card right at the action area (`app/integrations/page.tsx:451-479`) â€” one branch per `authType`:

- `'oauth2'` (Fortnox, Visma, TheFork) â†’ `<button onClick={() => connectFortnox()}>` (line 456)
- `'api_key'` (Personalkollen, Ancon, Caspeco, Trivec, Zettle, etc.) â†’ `<button onClick={() => { setModal(provider.key); setApiKey(''); setBizForPOS(selectedBiz) }}>` (line 461)

These two click handlers are the entire delta between the two flows.

## 2. PK connect flow (working pattern)

**Click handler** (`app/integrations/page.tsx:461`):

```tsx
<button className="btn btn-primary btn-sm"
        onClick={() => { setModal(provider.key); setApiKey(''); setBizForPOS(selectedBiz) }}>
  {t('actions.connect')}
</button>
```

The click does three things â€” and the third is the load-bearing one: **`setBizForPOS(selectedBiz)`** copies the currently-selected business id from page state into the modal's local state (`bizForPOS`) before opening the modal.

**Modal** (lines 528-541) further protects this: when the provider category is `'POS'` and the org has multiple businesses, the modal renders a `<select>` (line 534) that lets the user override `bizForPOS` per submission. So `bizForPOS` is set automatically to `selectedBiz` and the user can change it inside the modal if they need to.

**Submission handler** (`saveApiKey('personalkollen')`, lines 177-203):

```tsx
const saveRes = await fetch('/api/integrations/personalkollen', {
  method: 'POST', headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ action: 'connect', api_key: apiKey.trim(), business_id: bizForPOS }),
})
```

`business_id: bizForPOS` is in the body. The PK route on the server side (`/api/integrations/personalkollen`) writes the `integrations` row with that value populated, satisfying the upsert's `(business_id, provider)` unique key.

**Why it works:** the click-time copy `setBizForPOS(selectedBiz)` guarantees the value is always set to whatever the sidebar selection is at click time, and the modal exists as a place to override it before submitting. The modal also serves as the "ask for the API key" UX, but the business-id capture is the part that's specifically relevant here.

## 3. Fortnox connect flow (broken pattern)

**Click handler** (`app/integrations/page.tsx:455-459`):

```tsx
{!isConnected && provider.authType === 'oauth2' && (
  <button className="btn btn-primary btn-sm" onClick={() => connectFortnox()}>
    {t('actions.connect')}
  </button>
)}
```

`connectFortnox()` is called with **no argument**. The function signature accepts an optional `businessId` (line 138), but the click site does not pass it. So `businessId` is `undefined` from the very first line of the handler.

**`connectFortnox` body** (lines 138-146):

```tsx
function connectFortnox(businessId?: string) {
  track('integration_connected' as any, { provider: 'fortnox' })
  const url = businessId
    ? `/api/integrations/fortnox?action=connect&business_id=${businessId}`
    : '/api/integrations/fortnox?action=connect'
  window.location.href = url
}
```

When `businessId` is `undefined` (always true today), the URL drops the `business_id` query param entirely.

**Server-side fallout** (`app/api/integrations/fortnox/route.ts:102`):

```ts
businessId = searchParams.get('business_id') ?? ''
```

With no query param, `businessId` becomes `''` (empty string). It is then signed into the OAuth state HMAC payload at line 108. After the customer authorises at Fortnox and the callback fires, `verified.businessId ?? ''` resolves to `''` (line 157), and the upsert at line 215 writes `business_id: businessId || null` â†’ **NULL**.

The upsert's `onConflict: 'business_id,provider'` (line 224) cannot match an existing row whose `business_id` is NULL because Postgres treats `NULL = NULL` as `false` under unique constraints. Net effect: each Fortnox callback INSERTS a fresh row (or fails if a NOT NULL constraint were present, which it isn't on `business_id`), and downstream the `integrations` table holds zero usable Fortnox rows because the page's per-business reads (`allIntegs.find(i => i.business_id === selectedBiz)` at line 355) won't match a NULL row.

**Why it's broken:** there is no equivalent of `setBizForPOS(selectedBiz)` anywhere in the OAuth click path. The handler doesn't read `selectedBiz`, doesn't pop a modal, doesn't capture a business id at all. It assumes the URL will carry it; the URL doesn't.

## 4. The gap

The Fortnox `<button onClick={() => connectFortnox()}>` (line 456) is missing the same one-line state read that the PK `<button onClick={() => { setModal(provider.key); setApiKey(''); setBizForPOS(selectedBiz) }}>` (line 461) does. The PK path automatically wires `selectedBiz` (page state, kept in sync with the sidebar's business switcher via localStorage) into the request body via the modal's `bizForPOS` state. The Fortnox path skips that wiring entirely â€” it neither passes `selectedBiz` to `connectFortnox(...)` nor pops a modal where the value could be captured. Because `connectFortnox(undefined)` produces a URL without `business_id`, the OAuth state HMAC carries an empty business id, the callback writes `business_id: null`, the unique-constraint upsert can't dedup against an existing row, and the page's per-business lookup can't find the row anyway. The bug is structurally a single missing argument at one call site.

## 5. Recommended fix shape

Change line 456 from `onClick={() => connectFortnox()}` to `onClick={() => connectFortnox(selectedBiz)}` â€” the analogue of `setBizForPOS(selectedBiz)` on the api_key path. Add a guard so the button is disabled (or shows a helpful tooltip) when `selectedBiz` is empty so a user with no business selected can't start an OAuth flow that would land back in the same NULL-business state. Optionally â€” if the OAuth flow needs the same per-submission override the api_key modal provides â€” pop a tiny "which restaurant?" confirmation dialog before redirecting; that's a UX choice, not necessary for correctness.
