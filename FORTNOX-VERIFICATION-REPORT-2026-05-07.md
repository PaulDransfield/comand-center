## What the report will contain when run

`scripts/verification-report.ts` produces a markdown document with these sections (this stub mirrors the headings so you can see the planned shape):

### §1. Method
Date range, voucher count, request counts, duration, vouchers translated → periods.

### §2. Match summary
Counts of exact (< 0.01 SEK) / tolerable (< 1% AND < 100 SEK) / material drift across all (period × metric) comparisons. Per-period table showing API value, PDF value, delta, percentage, and bucket for every compared metric: `revenue`, `dine_in_revenue`, `takeaway_revenue`, `alcohol_revenue`, `food_cost`, `alcohol_cost`, `staff_cost`, `other_cost`, `net_profit`.

### §3. Tolerable drift detail
Per-period list of every tolerable-drift line so the advisor can see the magnitude of "good but not exact" matches.

### §4. Material drift findings
For each material-drift (period, metric) pair: the delta, the top API line items, the top PDF line items, and a one-line root-cause observation. Three observation patterns are pre-implemented:

- **PDF zero / API non-zero or vice versa** — points at one side missing data outright.
- **VAT subset mismatch** (`dine_in_revenue`, `takeaway_revenue`, `alcohol_revenue`) — voucher rows do not carry the "X % moms" label text Resultatrapport PDFs carry, so the API path's VAT split is structurally incomplete from voucher data alone. This is a known limitation acknowledged in the translation layer's header comment.
- **Account-coverage mismatch** — accounts present on one side and absent on the other, with the specific account numbers listed.

If none of those patterns match, the observation falls back to "same accounts on both sides — drift is amount-level, likely voucher-row rebookings between PDF cutoff and API fetch."

### §5. Implications for Phase 2
A fact-only paragraph stating one of three outcomes:

- **All exact:** Phase 2 can proceed as planned with high confidence.
- **All exact + tolerable, no material:** Phase 2 proceeds; UI should label API-derived data as "≤ 1% drift vs accountant report."
- **Material drift present:** Phase 2 cannot proceed as a like-for-like API replacement; recommended architecture is hybrid (API for daily/weekly tail, PDF apply for month-end reconciliation, both converging through `projectRollup`).

### §6. Out of scope (observed during this work)
- The two parallel sync paths (`lib/sync/engine.ts` vs `app/api/integrations/fortnox/route.ts`) — cleanup is a separate task.
- The `financial_logs` writes using `v.TransactionInformation` as a numeric amount — pre-existing bug, not Phase 1's job to fix.
- VAT-rate revenue split incompleteness from voucher data alone.
- Possibility that PDF-derived production data itself contains anomalies (M047 chokepoint is recent).

## What the harness can NOT determine — even when run

These are out of scope for Phase 1 and the report explicitly will not address them:

- **Whether the OAuth scopes currently requested (`bookkeeping invoice supplierinvoice salary companyinformation`) are sufficient for Phase 2.** `salary` and `companyinformation` are requested today but never used.
- **Whether Fortnox's pagination behaves as documented for the date range we're hitting.** The fetcher uses `page=1, 2, …` until `MetaInformation.@TotalPages` indicates the last page; if Fortnox's response shape diverges from the documented envelope, the fetch errors out loudly.
- **Whether the verification harness is correct.** Until the harness is run end-to-end and the report shows >0 exact matches against PDF data we know to be correct, the harness's correctness is asserted via typechecking and code review only.
- **Daily and dept-level metrics drift.** The harness doesn't run the aggregator. Daily and dept comparisons would need API-fetched POS data which is out of Phase 1 scope.

## Phase 2 readiness assessment (provisional)

This section is the only forward-looking content in the report and per the prompt's instruction is brief and clearly framed as observation.

**Provisional, pending harness run:** the audit (Question 1) already revealed two findings that affect Phase 2 scope independent of what the verification numbers say:

1. **Two parallel sync paths exist.** Path A (`lib/sync/engine.ts`) calls `/vouchers` and `/supplierinvoices` daily via cron and writes summary rows to `financial_logs`, but **nothing reads `financial_logs`**. Path B (`app/api/integrations/fortnox/route.ts`) writes directly to `tracker_data` bypassing the canonical `projectRollup` writer. Phase 2's "API-driven onboarding" needs to pick one path, retire the other, and route through `projectRollup`. This is a refactoring decision that pre-empts the question of whether API data matches PDF data.

2. **Voucher line-level detail is not currently captured.** Path A calls `GET /3/vouchers` (the list endpoint) only, and stores `v.TransactionInformation` (a free-text field) as if it were a numeric amount. The fetcher built in this session adds the `GET /3/vouchers/{series}/{number}` per-voucher detail call — that's new code, not productionised. Phase 2 productionisation needs this addition.

**If the harness run shows exact or tolerable match:** Phase 2 proceeds as planned. The voucher fetcher + translation layer in this session are the seed of the production fetcher; they need state tracking, resume on partial failure, and a customer-facing "Connect Fortnox" flow added on top.

**If the harness run shows material drift:** Phase 2 needs to address whichever of the §4 observations applies. The most likely material finding given the audit's structural observations is the VAT subset mismatch — the translation layer cannot recover dine_in/takeaway/alcohol from voucher row data without the chart of accounts (3xxx subranges by VAT class). If that's the dominant finding, Phase 2 needs an additional `GET /3/accounts` call at backfill time to derive the VAT mapping per customer.
