**Key observation:** every field the proposed layout needs is already in this payload. Three streams per day:

| Proposed row | Field in current payload |
|---|---|
| (1) Demand forecast | `suggested[i].est_revenue` (12-week pattern, weather-refined) |
| (2) Operator scheduled | `current[i].hours`, `current[i].est_cost` |
| (3) AI suggested | `suggested[i].hours`, `suggested[i].est_cost` |

The `est_revenue` row is the model's expected demand. **There is also a NEWER demand-forecast endpoint** at `/api/weather/demand-forecast` (shipped commit `0a83294` earlier today, 2026-05-07) which produces a more sophisticated per-day predicted-revenue with confidence label + holiday gate + recommendation text. The new endpoint and the scheduling agent's `est_revenue` use overlapping but not identical methodology — both anchor on per-weekday baseline × weather-bucket lift, but the demand-forecast endpoint adds confidence sample-size tagging and an explicit holiday gate.

## 4. Underlying data streams

### (a) Demand forecast per day

- **Lives in:** computed live by `/api/scheduling/ai-suggestion` (in `est_revenue` field, derived from 12-week per-weekday × weather-bucket pattern). ALSO computed by the newer `/api/weather/demand-forecast` (shipped today, 2026-05-07), which produces a more thorough predicted-revenue + confidence + holiday gate.
- **Storage:** neither endpoint persists the forecast. Both compute on every call. The newer endpoint has a 5-minute server-side cache table `overhead_drilldown_cache` is for overheads, NOT for demand-forecast — demand-forecast results are NOT cached.
- **Granularity:** per-day. No per-shift split today.
- **Available for Vero?** Yes once the OAuth completes Saturday and the 12-month backfill has populated `daily_metrics`. The historical correlation needs `weather_daily` to be populated too — that requires a one-time admin backfill (`POST /api/admin/weather/backfill?business_id=...`) which has not been run yet. Without `weather_daily`, the bucket-refined forecast falls back to plain weekday averages.

### (b) Operator's scheduled hours (per day, week ahead)

- **Lives in:** Personalkollen API (live fetch via `getWorkPeriods` — `lib/pos/personalkollen.ts`). Cached fallback in `staff_logs` table where shifts have `pk_log_url LIKE '%_scheduled'`.
- **Storage:** `staff_logs` table is the persisted form (synced daily via `runSync`). Live fetch in the ai-suggestion endpoint preferred because `staff_logs` may lag by 24h.
- **Granularity:** per-shift in the source; aggregated to per-day + per-department in `current[i].dept_breakdown`.
- **Available for Vero?** Yes — Vero already has an active PK integration. This stream has been working since onboarding.

### (c) AI's suggested hours (per day)

- **Lives in:** computed live by `/api/scheduling/ai-suggestion` in the `suggested[i].hours` field. Pure deterministic math — no LLM call, no Anthropic spend on this path.
- **Storage:** acceptances persist to `schedule_acceptances` (M026) when the user clicks Apply. The unaccepted suggestion itself is recomputed every call.
- **Granularity:** per-day. The model's full-precision target is in `model_target_hours`; the asymmetric cap (never above `current.hours`) yields `hours`. Per-shift breakdown is not produced.
- **Available for Vero?** Yes — needs ≥3 historical samples for the matched (weekday, weather-bucket); falls back to plain weekday-only averages when the bucket subset is too thin. With Vero's PDF-derived months of revenue data, this stream is live today.

**All three streams are queryable from a single API call (`/api/scheduling/ai-suggestion`).** No new computation, no new storage required.

## 5. The gap

**Smallest delta:** the proposed three-row weekly grid + savings callout is purely a new UI component. Every data field it needs — demand forecast (`suggested[i].est_revenue`), scheduled hours (`current[i].hours`), AI-suggested hours (`suggested[i].hours`), and the kronor savings (`summary.saving_kr`) — is already in the response body of `/api/scheduling/ai-suggestion`. There is no agent-output refactor required, no new database table, no new endpoint, no new computation. The existing UI rendering today (`AiHoursReductionMap`) is a per-day comparison of *two* of the three rows (current vs AI hours) using a stacked-bar visualization that emphasises the per-day cut decision; the proposed layout is the same data plus an explicit demand-forecast row above the schedule rows, and a more compact week-grid layout. One open architectural choice worth flagging: whether the demand-forecast row should be sourced from the existing `suggested[i].est_revenue` (lives inside the same payload, simplest plumbing) or from the newer `/api/weather/demand-forecast` (richer — confidence labels, holiday gate, dedicated recommendation text — but requires a parallel fetch).
