### 2. The two known pathways — confirmed details

#### `/api/scheduling/ai-suggestion` (#1)

- **No LLM call in the suggested path.** Every number — `est_revenue`, `rev_per_hour`, `model_target_hours`, `delta_hours`, `delta_cost`, the prose `reasoning` — is built from deterministic statistics in `app/api/scheduling/ai-suggestion/route.ts:217-403`. The file's `// @ts-nocheck` and the "AI" branding are misleading; the only AI-shaped artefact is the rationale string built by a switch over the same numbers. There is no `import Anthropic`, no `claude.messages.create`, no `ai_request_log` write inside this route.
- **Inputs.** The route fetches:
  - PK `WorkPeriods` live (when token present) for "current" schedule (`route.ts:80-116`). Falls back to `staff_logs` rows tagged `pk_log_url LIKE '%_scheduled'`.
  - Last 12 weeks of `daily_metrics` (`date, revenue, staff_cost, hours_worked, labour_pct`) ending the day BEFORE the requested range so the prediction never sees its own target window (`route.ts:164-167`).
  - Observed `weather_daily` rows over the same 12-week window (joined for bucket-aware refinement).
  - Live `getForecast(lat, lon)` — Open-Meteo (cached 1h in-process).
  - Same-period `daily_metrics` rows with `revenue > 0` for the this-week pull-forward scaler.
- **Math.**
  1. Group history by weekday and by (weekday × weather-bucket).
  2. For each target date, raw `avgRev = weightedAvg(srcRev, srcDates)` from `lib/forecast/recency.ts` — last 28 days count 2× older entries.
  3. Compute median ratio of completed-day actuals to the model's raw prediction in the SAME range; clamp to [0.75, 1.25] (`thisWeekScaler`). Multiply remaining-day predictions by it.
  4. Target hours = `avgRev / p75(rev_per_hour)`. Clamp to ≤ current scheduled hours (asymmetry rule — never recommends adding).
  5. Delta cost = `deltaHours * (current.est_cost / current.hours)`.
- **Cost / latency.** Zero AI cost. Latency is dominated by the live PK fetch + the 4 parallel Supabase queries; well under the 60s `maxDuration`. No `ai_request_log` row is written.
- **Cache.** None — `dynamic = 'force-dynamic'`, `noStore()`, response headers `Cache-Control: no-store, max-age=0, must-revalidate`. The response is recomputed on every dashboard load. Open-Meteo's forecast is in-process cached 1h inside `lib/weather/forecast.ts`.
- **Fallbacks for malformed/empty output.** When PK live fetch fails or returns 0 periods, the route falls back to `staff_logs` rows with `pk_log_url LIKE '%_scheduled'` (`route.ts:118-131`). When fewer than 3 historicals exist for a (weekday) or (weekday × bucket) combination the rationale string degrades to "not enough signal — holding your schedule". `est_revenue` for a date with no positive history rows simply lands as 0 and the day renders that way.

#### `lib/weather/demand.ts` (#2)

- Confirmed as the primary computation; `/api/weather/demand-forecast` is a thin wrapper (40 lines) that calls `computeDemandForecast`.
- **Math, step by step (`lib/weather/demand.ts:100-264`):**
  1. Load business (`businesses.id, org_id, name, city, country`); resolve coords via `coordsFor(city)`.
  2. In parallel: fetch `getForecast(lat, lon)` (Open-Meteo, up to 16 days), fetch bucket lifts from a 365-day join of `weather_daily × daily_metrics`, fetch baseline-by-weekday from last 12 weeks of `daily_metrics`, fetch this-week actuals.
  3. **Bucket lift** (`fetchBucketLifts`): per-bucket recency-weighted avg revenue ÷ overall recency-weighted avg revenue. Returns `factor` and `sampleSize` per bucket (clear / mild / cold_dry / wet / snow / freezing / hot / thunder).
  4. **Baseline** (`fetchBaselineByWeekday`): last 12 weeks of `daily_metrics`, group by `weekday`, recency-weighted avg of `revenue` where `revenue > 0`.
  5. **First pass** loops the forecast and computes `rawPredicted`. Holiday → `baseline`. With lift + baseline > 0 → `baseline × lift.factor`. Else → `baseline`.
  6. **This-week scaler**: pair completed days' actuals to `rawPredicted` for dates in the current Mon–Sun. `thisWeekScaler` returns the clamped median ratio.
  7. **Second pass** writes `predicted_revenue`. If the date is a holiday → `baseline` (no lift). Else if lift+baseline → `baseline × lift.factor`. Else → `baseline`. Then if the date is in the current calendar week, has no logged actual yet, and isn't a holiday — multiply by `weekScale`.
  8. `delta_pct = (predicted − baseline) / baseline × 100`. `confidence`: `high` if bucket sampleSize ≥ 8, `medium` ≥ 4, `low` else; `unavailable` when no baseline at all.
  9. `recommendation` (one-liner, gated to confidence ≥ medium, |delta| ≥ 12%, non-holiday) is hand-coded prose driven by bucket + isWeekend.
- **Zero LLM in this pathway.** No Anthropic import; no `ai_request_log`. The output is consumed verbatim by the widget and embedded as a plain-text block in the Monday Memo prompt (#5).
- **Holiday handling (confirmed).** When `is_holiday = true`, `predicted_revenue = baseline` (the bucket multiplier is bypassed) and `confidence = 'low'`. Holidays are NOT scaled by `weekScale` either. The widget renders these distinctly. The dashboard investigation's claim that holidays return `baseline_revenue` "as-is" matches the code at `lib/weather/demand.ts:192-196`.

## Section C — Reconciliation gap

### 8. Is reconciliation logged anywhere?

**Partial. There is a real reconciliation pipeline, but it covers only the BUDGET surfaces and only at MONTHLY granularity. The two daily revenue forecasters that drive the dashboard chart and DemandOutlook widget (#1 and #2) are NOT reconciled against actuals anywhere — no table stores "we predicted X, actual was Y" for them.**

Evidence:

1. **`ai_forecast_outcomes` (M020) IS the audit table.** Schema at `archive/migrations/M020-AI-FORECAST-OUTCOMES.sql`. It has `suggested_*`, `actual_*`, `revenue_error_pct`, `revenue_direction`, `staff_cost_error_pct`, `margin_error_pp`, `owner_reaction`. The `surface` CHECK constraint enumerates: `'budget_generate' | 'budget_coach' | 'budget_analyse' | 'weekly_memo' | 'tracker_narrative'`. Notably absent: `'scheduling_ai_revenue'`, `'weather_demand'`, `'anomaly_baseline'`, `'monday_memo_action_impact'`.

2. **Capture surfaces today**: `app/api/budgets/generate/route.ts:598-605` writes `surface = 'budget_generate'` rows; `app/api/budgets/coach/route.ts:128-147` calls `captureOutcome` with `surface = 'budget_coach'`. **No other capture sites exist** — `grep` for `captureOutcome` finds 2 files (`outcomes.ts` itself + the coach route); `grep` for `ai_forecast_outcomes` writes finds the same plus `budgets/feedback/route.ts` (owner reaction update) and `cron/ai-accuracy-reconciler` (the resolver). `weekly_memo` and `tracker_narrative` are valid `surface` values per the CHECK but are NEVER written.

3. **Reconciler** (`app/api/cron/ai-accuracy-reconciler/route.ts`) runs daily at 07:00 UTC. It walks unresolved rows whose period has closed, joins `monthly_metrics` for that `(business_id, year, month)`, fills in `actual_revenue / actual_staff_cost / actual_food_cost / actual_other_cost / actual_net_profit / actual_margin_pct`, computes `revenue_error_pct`, classifies direction (`accurate` if |err| < 5 %, else `under` / `over`). Confirmed working: Vero has 5 resolved rows (Mar/Apr 2026 budget_generate + budget_coach) with real `revenue_error_pct` values (e.g. budget_generate Feb 2026 underpredicted by 41.3 %).

4. **Feedback loop closes the prompt**: `app/api/budgets/generate/route.ts:138-147` reads the last 12 months of resolved outcomes and feeds them into the next prompt. The Vero data shows this loop is firing — owner has marked 12 of those rows `too_high`.

5. **What is NOT logged anywhere**:
   - `/api/scheduling/ai-suggestion`'s `est_revenue` is never compared to actuals. There is no `scheduling_forecast_log` or similar table; the `daily_metrics.updated_at` overwrites the row when actuals land but doesn't preserve the prediction. Once the date passes, the forecast is gone — only `briefings.key_metrics` retains a snapshot for the week the memo covered, and that's a summary aggregate, not the per-day prediction.
   - `/api/weather/demand-forecast`'s `predicted_revenue` is similarly ephemeral. The output is computed on every request from current state; nothing persists `(date, predicted_revenue, actual_revenue)`.
   - Anomaly detection logs `metric_value, expected_value, deviation_pct` on each `anomaly_alerts` row — that is technically a "we expected X" audit, but there is no resolution column ("was this a true positive?") and no UI for owners to confirm/dismiss with feedback. `is_resolved` exists on the table (per `app/alerts/page.tsx`-side queries) but resolution doesn't reconcile against future actuals.
   - The Monday Memo's 3 actions each carry an SEK-impact prediction. The `briefings` table stores the `content` and `key_metrics` JSON, but no follow-up reconciliation cron compares "promised +4 200 kr/wk" to the realised next-week delta.

6. **Closest-existing-thing for daily revenue forecasts**: `briefings.key_metrics.revenue` (last week's total) is preserved per memo, so a week-over-week prediction error COULD be derived if a new pipeline read consecutive `briefings` rows. But this is summary-level, weekly, and was never designed as a prediction-vs-actual ledger.

7. **`forecast_calibration` is not the same thing**. It's a per-business single-row table (`UNIQUE(business_id)`) holding `accuracy_pct`, `bias_factor`, `dow_factors`. It calibrates only the legacy `forecasts` table (whose `revenue_forecast` per (business, year, month) is straightforward arithmetic, not LLM-driven). Vero's row: `accuracy_pct = 82`, `bias_factor = 1.1`, `dow_factors = {Mon: 0.18, Tue: 0.34, Wed: 0.55, Thu: 0.57, Fri: 2.31, Sat: 2.79, Sun: 0.009}`. Note the Sun value is plainly wrong — Vero is closed-ish on Sundays so the 0.009 is a sample-size artefact; the cron writes whatever the 90-day average produces with no minimum-sample guardrail. The legacy `forecasts` table's predictions (Vero has 12 rows there) are not in `ai_forecast_outcomes` and not reconciled by the M020 reconciler.

## Section D — Data quality

### 11. Vero's actual data state

- **`daily_metrics`**: 706 rows from 2024-02-01 to 2026-05-08. Of those:
  - 145 rows have `revenue > 0` (first positive day **2025-11-24**, when PK was wired up).
  - 561 rows have `revenue = 0` — these are the 2024-02 → 2025-11 period where staff cost was loaded (from PK historical) but no POS provider was emitting revenue yet. Forecasters skip them via `WHERE revenue > 0`, so they don't poison baselines, but the volume of zero rows means any naive "12 weeks of history" query that doesn't filter will dilute averages.
  - **Covers = 0 on every single one of the 706 rows.** No POS integration emits covers for Vero. Anything in the prediction stack that depends on covers is dead-code for this customer.
  - `labour_pct` is null on 572/706 rows. Only 134/706 (~19 %) have both `revenue > 0` AND non-null `labour_pct`.
  - 124 dates are missing entirely inside the [2024-02-01..2026-05-08] window — explained mostly by the period before PK was synced; checked by enumerating dates, ~all gaps fall before 2025-11-24.
  - `(rev_source, cost_source)` distribution since 2025-11: `pos|pk` 134, `none|pk` 21, `pos|none` 11. The 21 `none|pk` days are days where PK pushed staff cost but no POS-side revenue arrived — a forecast trained on these days would over-predict labour %.
- **`monthly_metrics`**: 28 rows, contiguous Feb 2024 → May 2026. 2024 months all `revenue = 0` (the 2024 `staff_cost` confirms business was operating). 2025-Q1 onward has revenue (Fortnox PDFs filled the gap that POS couldn't). 2026-04 and 2026-05 have `food_cost = 0` because Fortnox PDFs for those months haven't been uploaded yet — `monthly_metrics` shows partial picture.
- **`tracker_data`**: 21 rows. 6 manual 2024 rows (rev=0). 15 `fortnox_pdf` rows for 2025-01 → 2026-03 inclusive — **continuous** (no Sep/Oct gap on Fortnox's side; the gap mentioned in `project_vero_2025_data_gap.md` is between Fortnox totals and POS daily-grain data, NOT inside `tracker_data`). The 2025 totals here imply ~12.7M revenue for the year, reconcilable against the accountant.
- **Anomalies firing for Vero right now**: 7 of the 11 most recent are `ob_supplement_spike` HIGH/CRITICAL on consecutive days 2026-05-02 → 2026-05-08 (deviations 69-84 %). The 7-day-vs-28-day comparison is firing every day because the OB supplement is consistently elevated. This is a baseline-too-stale problem — once a new normal sets in, the rolling baseline should adapt, but the detector compares 7d to 28d anchored on the prior 28 days, so it'll keep firing as long as the recent week is materially above the older 4 weeks. Owners reading the alerts list see 7 critical-ish alerts about the same thing.
- **Day-of-week factor sanity**: `forecast_calibration.dow_factors` for Vero has Sun = 0.009 — that's a sample-size artefact (Vero is mostly closed Sundays). Any consumer using these factors without a sample-size check would predict Vero to do ~90 kr on a Sunday.
- **Zero-revenue / staff>0 contamination**: 11 days since 2025-11 are tagged `(pos, none)` — POS reports revenue but PK has no shifts logged. These are valid for revenue prediction but contaminate any labour-% prediction.
- **Anomaly contamination of baselines**: anomaly detection writes to `anomaly_alerts` but does not flag the underlying day on `daily_metrics`. The forecasters do not exclude anomaly-flagged days when computing baselines. So a 2× outlier day passes straight into the rolling average.
- **Suspicious zero values**: 4 days with `revenue < 5 000` since 2025-11 (e.g. 2026-05-04 rev=4 399) survive the `WHERE revenue > 0` filter and lower the weekday average for that day's bucket.

### 12. Schema history

Recent prediction-relevant schema changes (per `MIGRATIONS.md` and the SQL files in `sql/`):

- **M008 (daily_metrics + monthly_metrics + dept_metrics)** — original schema. Has not been altered. Columns and types as listed in §4.
- **M015 (weather_daily)** — file at `sql/M015-weather-daily.sql`. **Not currently visible in PostgREST schema cache for this project** (live diagnostic returned `PGRST205`), so either the migration was never applied or the table was dropped. Either way, `lib/weather/forecast.ts` reads/writes assume it exists; both forecasters degrade silently when the table is missing. This is the single biggest data-quality issue affecting the prediction stack.
- **M020 (ai_forecast_outcomes)** — adds the audit table and the `prune_ai_forecast_outcomes` function. Loaded; verified via Vero having 13 rows. Columns include `suggested_*`, `actual_*`, `revenue_error_pct`, `revenue_direction`, `owner_reaction`. The CHECK on `surface` is the surface-allowlist mentioned in §8.
- **M046 (businesses.opening_days + business_stage)** — present on Vero's row (`opening_days` = all 7 days; `business_stage` = null). The forecasters do NOT yet read `opening_days` to skip closed-day predictions (Vero's Sunday DoW factor of 0.009 is a direct symptom).
- **M047 (Fortnox apply guardrails) + tracker_data.created_via** — new column on `tracker_data`. Pre-M047 rows have `created_via = null`; post-M047 are tagged. All 21 of Vero's tracker rows show `created_via = null`, which means none of them went through the validated chokepoint — they're either pre-M047 inserts or the column wasn't backfilled.
- **M032 (fortnox_supersede_links)** — affects re-uploads but doesn't change the shape of `tracker_data` rows, so a learning system reading the historical `tracker_data` is safe as long as it filters by `status = 'applied'` on the corresponding `fortnox_uploads` row.

Specific column-rename / semantic-shift risks for a future learning model:
- The `daily_metrics.cost_source` enum is documented in CLAUDE.md Session 16 as growing new values (`fortnox_pk_disagrees`, `fortnox_pk_partial`, etc.) — a model that one-hot-encodes this column needs to be re-trainable as new codes appear.
- `tracker_data` had pre-Session-13 entries that did not follow the new sign convention from `lib/finance/conventions.ts`. The single-writer rule (`projectRollup`) is recent; older rows may have their `net_profit` computed under a different convention. A learning system reading `net_profit` from rows older than ~Session 12 should re-derive it from components.

## Summary — what a learning prediction system would need to be built

Eleven prediction pathways live in the codebase today; only two of them (#1 scheduling-AI revenue forecast, #2 weather-demand) drive numbers the owner sees on the dashboard, and both are pure math, not LLM. The two paths share `lib/forecast/recency.ts` (added today, commit `f6029ca`) but otherwise duplicate weekday-baseline + this-week-scaler logic — they would consolidate cleanly. Five LLM-touched surfaces emit forward-looking claims (Monday Memo's 3 actions, budget generate, budget coach, budget analyse, tracker narrative); only `budget_generate` and `budget_coach` are reconciled, via the existing `ai_forecast_outcomes` table (M020) at MONTHLY granularity. The dashboard's daily revenue forecast (`est_revenue`) and the weather-demand widget's `predicted_revenue` are NOT in the audit table — there is no log of "we predicted X for 2026-05-04, actual was Y". The reconciliation infrastructure that DOES exist — `ai_forecast_outcomes` + the daily 07:00 UTC `/api/cron/ai-accuracy-reconciler` cron + the resolved-rows feedback into the next budget prompt — is the closest existing scaffold; a parallel daily-grain ledger (capturing each daily prediction at generation time, joining `daily_metrics` next morning) would be the minimal slice. Signals currently usable for cross-customer learning: `daily_metrics` (revenue, staff_cost, labour_pct, hours, shifts), `monthly_metrics`, `staff_logs`, `holidays/<country>.ts` pure-compute, Open-Meteo forecast + (when populated) `weather_daily`. Signals available for any new customer with no work: holidays, Open-Meteo, day-of-week, day-of-month. Signals derivable but not yet leveraged: year-over-year same-MONTH (we have 16 months of `monthly_metrics` for Vero), klämdag-style proximity-to-holiday, salary cycle (25th of month), recent-trend acceleration. Substantial-new-infra signals: school-break calendar, local events, school-holiday vacations. Vero's data quality realities constrain what learning can do today: of 706 `daily_metrics` rows, only 145 have positive revenue (first day 2025-11-24); 0 have covers; 134 have positive revenue AND non-null labour_pct; the `weather_daily` table appears to be missing from prod entirely (PGRST205) so the bucket-lift refinement degrades to baseline-only; `forecast_calibration.dow_factors` for Vero has Sun = 0.009 which would corrupt any consumer that multiplies by it. The customer-base reality is one real customer (Vero) with ~5.5 months of dense daily data; cross-customer training is not viable yet. Any architecture must (a) build the reconciliation ledger first, with a daily-grain extension of M020's pattern, (b) separately fix the missing `weather_daily` table since two of three current forecasters silently degrade without it, and (c) plan for "1 customer learning from itself" as the realistic regime for the next 6+ months — a framework that learns cross-customer once 5-10 customers are live, but doesn't break with one.
