## 2. The big chart

- Default export: `OverviewChart`
- **Charting library: hand-rolled SVG.** No Recharts, Chart.js, or D3. Fixed viewBox (`660×320`), `<rect>` per bar, `<line>` per whisker, `<path>` per gross-margin line segment. Tooltip is a positioned absolute `<div>` (`ChartTooltip` sub-component).

### Data shape

Consumes a `DayRow[]`:
```
date, revenue, staff_cost, staff_pct (=labour_pct), dateStr, dayName,
dayIdx?, isToday, isFuture, hasActualData?, isClosed?,
pred?: { est_revenue, planned_cost, ai_cost, delta_cost, weather, bucket_days, under_staffed_note },
prevDay?: { revenue, staff_cost }
```

Granularity is daily. Either 7 points (week mode) or `daysInMonth` (28-31 in month mode).

### Source endpoints

Revenue + labour come from the **same** endpoint per period — `/api/metrics/daily` reads `daily_metrics` rows (one per business per day), pre-aggregated by `lib/sync/aggregate.ts`. Forecast comes from a separate endpoint, `/api/scheduling/ai-suggestion`.

### Compare modes (None / Prev / AI)

Implemented as horizontal dashed whiskers across each bar (not separate bar series):

- **none** → no whisker.
- **prev** → use `d.prevDay.revenue` / `d.prevDay.staff_cost` (built page-side from the previous-period `/api/metrics/daily` call). Only shows whiskers for days where the previous period actually had data.
- **ai** → use `d.pred.est_revenue` / `effectiveAiCost(d)`. `effectiveAiCost` enforces the asymmetric "cuts only" policy — when `under_staffed_note` is true the AI returns the planned cost rather than recommending an increase.

So "AI" comes from the **same** `/api/scheduling/ai-suggestion` endpoint that drives the labour scheduling card, not a separate forecast service.

### Predicted bars for upcoming days

Yes — the chart distinguishes actual vs predicted at the day level:

- Past/today actuals: solid navy revenue bars + solid burnt-orange labour bars.
- Future days (or current day before sync): same bar position, but filled with a 45° striped pattern (`pk-pred` for revenue, `pk-pred-lab` for labour) and a soft outline. Reads as "soft / forecast" against the solid actuals.
- Gross-margin line: split into segments by `dayMarginKind` — solid green on `real` segments (real or closed), dashed green on `pred` segments. Markers are filled circles for real, hollow rings for predicted.
- Closed past days (no `daily_metrics` row at all) draw the margin line at zero rather than inheriting the AI forecast (FIXES §0xx).

### Anomaly markers on the chart

**No.** The chart does not annotate anomalies inline. The only inline highlight is the X-axis day label colour (red for weekends and public holidays, green-bold for today). The anomaly banner (top of page) is an entirely separate row above the chart.

## 4. Anomaly detection — the OB-tillägg case

### The detector exists. It's specifically the OB supplement.

- File: `C:\Users\Chicce\Desktop\comand-center\lib\alerts\detector.ts:346-387`
- The OB-tillägg check is one of four signal types in a general anomaly detection system. It is **not** a generic system that incidentally flagged OB — there is a dedicated check that reads `staff_logs.ob_supplement_kr`, compares the last 7 days' average per shift against the previous 28 days' average per shift, and emits `alert_type='ob_supplement_spike'` when the recent average is ≥ 40% above the historic average.
- Title format (line 380): `` `OB supplement spike +${obDev.toFixed(0)}% — ${bizName}` ``. This is byte-for-byte the banner string in the screenshot.
- Severity buckets: medium ≥ 40%, high ≥ 60%, critical ≥ 80%.
- Description template (line 381): `OB supplement is {recentAvg} kr/shift vs {historicAvg} kr/shift average. Check for unplanned overtime.` — the banner text "Your supplements cost per cover jumped on May 8th — likely due to ei…" is **not** the raw template; it has been rewritten by `explainAnomalyDescriptions()` (lib/alerts/detector.ts:30-83), which posts each alert to Claude Haiku with a one-sentence "explain to a restaurant owner" prompt and overwrites `alert.description` with the AI's response. So the description prose is generative; the title and the metric are deterministic.

### What categories does the system cover

Two detector libraries write into the same `anomaly_alerts` table:

1. `lib/alerts/detector.ts` — monthly-level detectors (run once per cron):
   - `revenue_drop` (≤ −15% MoM vs 4-month average)
   - `food_cost_spike` (≥ +5pp food cost % vs 4-month avg)
   - `staff_cost_spike` (≥ +8pp staff cost % vs 4-month avg)
   - `ob_supplement_spike` (the OB-tillägg detector — ≥ +40% per-shift recent vs historic)

2. `lib/alerts/line-item-anomalies.ts` — Fortnox other-cost line item detectors:
   - `new_vendor` (label appears for first time in 6 months)
   - `cost_creep` (>15% MoM rise for 3 consecutive months)
   - `spike` (one-month amount > 3× rolling 5-month median)
   - `dormant` (subscription disappears for 2+ consecutive months)

Both libraries run daily from `app/api/cron/anomaly-check/route.ts` at 05:30 UTC (vercel.json line 12-14). Both write to the same `anomaly_alerts` table.

### Storage

Table: `anomaly_alerts` (M008-summary-tables.sql era; alternate column list in MIGRATIONS.md line 331).

Columns: `id, org_id, business_id, alert_type, severity, title, description, metric_value, expected_value, deviation_pct, period_date, is_read, is_dismissed, email_sent_at (M042), email_recipients (M042), created_at`.

Lifecycle:
- **Created**: by `runAnomalyDetection` / `runLineItemAnomalies` once per day per (org, business, alert_type, period_date) — dedup via existence check before insert (`detector.ts:198-206`).
- **Marked read**: PATCH `/api/alerts {id, action: 'mark_read'}` → sets `is_read=true`.
- **Dismissed**: PATCH `/api/alerts {id, action: 'dismiss'}` → sets `is_dismissed=true`.
- **Expired**: no explicit expiry — alerts persist until dismissed. The dashboard only shows alerts where `is_dismissed=false` AND `is_read=false`.

### Endpoint feeding the dashboard banner

`GET /api/alerts` (`C:\Users\Chicce\Desktop\comand-center\app\api\alerts\route.ts`)

Response shape: array of alert rows with a joined `businesses(name, city)` object.

## 5. Weather + holidays

### Weather source

`/api/weather/demand-forecast?business_id&days` (`app/api/weather/demand-forecast/route.ts`) is what the dashboard's `WeatherDemandWidget` reads. Internally it calls `computeDemandForecast` in `lib/weather/demand.ts`, which:

- Fetches a 7-day forecast via `getForecast()` in `lib/weather/forecast.ts`. The actual weather API is **Open-Meteo** (which sources from SMHI for Sweden); the file's header confirms this.
- Bins each day into a weather bucket (`clear`, `mild`, `cold_dry`, `wet`, `snow`, `freezing`, `hot`, `thunder`).
- Multiplies a per-business per-weekday rolling baseline (8-12 weeks of `daily_metrics`) by a learned per-bucket lift coefficient → `predicted_revenue`.
- Tags each day with `is_holiday` and `holiday_name` from `lib/holidays`.

So the weather endpoint already returns `predicted_revenue` per day and joins holidays inline.

### Per-day predicted revenue — same source as scheduling agent's `est_revenue`?

**No, they're different.** The chart's per-day forecast uses `aiSched.suggested[].est_revenue` from `/api/scheduling/ai-suggestion`. That endpoint computes its own forecast (via PK history × weather buckets) inside the scheduling AI route. The weather widget's `predicted_revenue` comes from `/api/weather/demand-forecast`. Both rely on weather buckets and per-business rolling baselines, but they are independent code paths. There is no shared "predicted revenue per day" service — same intuition, two implementations.

The dashboard currently uses **only the scheduling AI forecast on the chart** and **only the weather forecast in the bottom strip**. They are not cross-referenced.

### Holidays source

Pure compute, no DB. Files:

- `lib/holidays/index.ts` — country router with `getHolidaysForCountry(country, year)` and `getUpcomingHolidays(country, fromDate, daysAhead)`.
- `lib/holidays/sweden.ts` — Anonymous Gregorian Easter algorithm + fixed dates + "first weekday in window" for Midsummer / All Saints'. 17 SE holidays per year.
- `lib/holidays/norway.ts`, `lib/holidays/uk.ts` — sibling country modules. Active.

Endpoint: `GET /api/holidays/upcoming?days=N&business_id=X` (`app/api/holidays/upcoming/route.ts`). Resolves country from `businesses.country` (default `SE`), returns localised name (`name_sv` / `name_en`) plus `days_until`.

Country code is sourced from `businesses.country` per business. The dashboard page itself doesn't call `/api/holidays/upcoming` — it imports `getUpcomingHolidays` from `lib/holidays` directly into the client bundle and computes upcoming holidays for the AttentionPanel client-side. The OverviewChart receives `holidayDateSet` (a `Set<string>`) computed in `app/dashboard/page.tsx:521-542` for X-axis red-label rendering.

The scheduling agent's "holiday gate" mentioned in earlier investigations is the same compute — `lib/weather/demand.ts` imports `getUpcomingHolidays` from `lib/holidays`.

### Joinability for the redesign's combined day cards

Yes — this is essentially already done by `/api/weather/demand-forecast`. Each `DemandDay` already carries:

```
date, weekday, weather { summary, temp_min, temp_max, weather_code, bucket },
is_holiday, holiday_name, baseline_revenue, predicted_revenue,
delta_pct, confidence, sample_size, recommendation
```

So one fetch (`/api/weather/demand-forecast?days=7`) is sufficient to render "Wednesday May 13 · sunny 14°/19° · 28 900 kr forecast · holiday Kristi himmelsfärds dag" in the v6 mockup's day cards. The shape is compatible.

The one nuance: revenue forecasts on holidays are not reliable per `lib/weather/demand.ts` — the model returns `baseline_revenue` (not `baseline × bucket_lift`) and the `WeatherDemandWidget` displays `t('holiday')` instead of a delta percentage. The redesign should keep that behaviour or replace it with explicit "holiday" labelling, not pretend the prediction is high-confidence.

Per day. The current chart already draws a labour bar (below the zero line) for every day. They are uniformly burnt-orange and have no above-bar text. Adding above-bar percentages and traffic-light colours is purely a presentational layer over data that's already in the DOM.

### Forecasted labour cost for future days

Yes — there are two parallel forecast streams:

1. `aiSched.current[date].est_cost` — what the **current schedule in PK** says will cost (sum of scheduled hours × hourly rate).

## 7. The "this week" stats on the chart header

### Margin

- Computed page-side: `margin = Math.max(0, totalRev - totalLabour)`.
- Note this is **gross margin minus labour only**. It is NOT net margin (food, other_cost, depreciation, financial are not subtracted). The `OverviewHero` footer shows it as "X kr margin"; the OverheadReviewCard separately shows true net margin from `tracker_data` for the latest applied month.

- Coverage caveat: covers data is only as good as the staff entering it at the till. Many small / takeaway-only restaurants won't have meaningful covers. Vero / Rosali both have it (PK is the source).

The dashboard does NOT currently display covers prominently on the dashboard page itself — it's available in `summary.total_covers` from `/api/metrics/daily` but the page never reads it. Surfacing covers on the chart header would use a value that already exists in the loaded payload; no new endpoint required.

## 10. Departments

- Source: `/api/departments` (`app/api/departments/route.ts`).
- Department definitions come from two merged sources:
  1. The `departments` table (admin-configured: `name`, `color`, `sort_order`). Optional — may not exist for every org.
  2. Distinct values of `staff_logs.staff_group` (always available from PK sync). Fills gaps when the `departments` table is empty / partial.
- Revenue per department is obtained by looking up `revenue_logs` rows whose `provider` matches `inzii_<slug>` or `pk_<slug>`. The `inzii_*` provider was the direct Swess/Inzii API path (now dropped — see memory `project_inzii_dropped.md`); `pk_*` is the per-workplace Personalkollen path (the canonical source today).
- Staff cost per department is obtained from `staff_logs.cost_actual` / `estimated_salary`, grouped by `staff_logs.staff_group`.

### Why the card was empty for Vero (Week 19)

Two compounding reasons:
1. PK per-workplace revenue (`pk_<dept>` providers) is only present in `revenue_logs` if PK is configured to sync per workplace AND the workplaces are mapped to dept names matching the `staff_group` values. Vero's earlier setup didn't have this clean per-workplace split — its revenue was syncing as the `personalkollen` provider (full-business aggregate), which is not matched by `deptToProviderKeys()` (which generates `inzii_…` and `pk_…` keys, not the bare `personalkollen` key).
2. `staff_logs.staff_group` may be present (so departments would be defined), but with no matching `pk_<slug>` rows in `revenue_logs`, `deptAcc[dept].revenue = 0` for every dept. Filtering in `DepartmentsSummary` removes departments where both revenue and staff_cost are 0 — depending on whether staff_logs are tagged, the card either shows the dept names with 0 revenue or shows the `departments.empty` string.

The screenshot's "No department data for this period yet" matches the empty branch — meaning Vero either has no `staff_group` values populated for the visible week, or both PK per-workplace revenue and staff dept tagging are absent for the period.

This is a configuration / data-availability issue, not a code bug. Other businesses with cleanly per-workplace PK setup (and correctly tagged staff_groups) populate the card correctly.

- The combined weather + holidays day card the mockup wants is already what `/api/weather/demand-forecast` returns: each day carries weather bucket + temp range + weather code + `is_holiday` + `holiday_name` + `predicted_revenue` + `delta_pct` + `recommendation` in one fetch. The mockup's day card shape is already expressible from one `DemandDay` row.

If the mockup wants every future day in the visible window to display a labour percentage, that's a small new fallback (e.g. baseline labour % from rolling N-week average), not a new endpoint.

## Recommended implementation-prompt shape

Splitting the prompt into (a) chart + cards visual reskin and (b) forecast-stream consolidation for the day cards is a safer cut than a single megaprompt.
